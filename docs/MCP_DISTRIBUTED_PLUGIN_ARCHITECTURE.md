# MCP 分布式插件架构方案

## 目标

本阶段为 Antigravity-Go (AGo) 增加一条可审计的 MCP 扩展链路，覆盖：

- 内核 MCP 写操作能力探测
- Web 端 MCP 配置与变更入口
- 智能体动态工具热刷新
- 工具命名冲突治理
- 外部进程安全边界控制

## 设计结论

### 1. 双层控制面

系统分为两层：

- 内核 RPC 层：负责真实的 MCP 注册、刷新、停止、重启与工具调用。
- 宿主管理层：负责 Web API、配置归并、动态工具注册和会话广播。

这样做的原因是把进程控制和 界面状态管理分离，避免前端直接耦合内核协议细节。

### 2. 写操作采用“探测 + 声明式刷新”

针对 `AddMcpServer`、`StopMcpServer`、`RestartMcpServer`，宿主侧提供显式 RPC 封装，并在运行时做方法探测：

- 若内核直接暴露目标 RPC，则记录为支持。
- 若未暴露，则保持能力报告为“不支持”，不伪造成功。

实际的“新增 / 更新 / 删除”路径优先走已知稳定的 `RefreshMcpServers` + `override_mcp_config_json`，原因：

- 该链路在当前仓库已有事实依据。
- 它是声明式配置，便于审计和回放。
- 它避免依赖尚未被本地 core 验证过的私有请求结构。

`Restart` 则优先尝试 `RestartMcpServer`，若不可用，降级为目标服务器级的 `RefreshMcpServers(server_name=...)`。该行为会在返回结果中明确标注 `operation_mode`，不是隐藏回退。

### 3. MCP 工具命名冲突治理

模型上下文中的工具名必须全局唯一，因此采用规范化命名：

`mcp__<server>__<tool>`

规则：

- 服务名与工具名都先转为小写。
- 非字母数字字符统一替换为 `_`。
- 若仍发生冲突，则追加稳定短哈希。

示例：

- `postgres` + `query` -> `mcp__postgres__query`
- `Postgres` + `query` -> `mcp__postgres__query`
- 两个不同服务规范化后同名 -> `mcp__postgres__query__a1b2c3d4`

这样可以保证：

- 模型工具列表无重名
- 日志、审计、会话回放中的工具名稳定
- 用户仍可在 界面中看到原始 server/tool 名称

### 4. 智能体热刷新策略

Web 服务维护一份基础 智能体和多份会话 智能体副本。MCP 配置发生变更后：

1. 先调用 Core 刷新服务器列表
2. 重新拉取 MCP server states
3. 重新生成动态工具定义
4. 用 `mcp__` 前缀批量替换基础 智能体中的旧动态工具
5. 对所有已连接 Web 会话做同样替换
6. 通过 WebSocket 广播 `mcp_tools_updated`

这样可以保证新工具立即进入后续对话的 模型工具上下文，不需要用户刷新页面或重开会话。

### 5. 外部进程安全性

安全边界如下：

- Web 默认仅监听回环地址，避免远程暴露。
- MCP 写操作必须显式通过 `/api/mcp` 触发，不在后台静默执行。
- 所有动态 MCP 工具默认要求人工授权，避免外部进程产生不可见副作用。
- 环境变量键名采用白名单校验，只允许合法 shell 标识符。
- 服务端对 `name`、`command`、`env` 做显式校验，失败直接返回错误，不做默认兜底。
- Host 不自行常驻托管额外外部守护进程，真实生命周期仍交给 Core，减少权限边界分叉。

### 6. 可观测性与可审计性

每次 `/api/mcp` 返回：

- 最新 server 列表
- 内核能力探测结果
- 本次操作的 `operation_mode`

这保证“能力是否真实存在”和“本次操作通过哪条链路生效”都可见。

## 已知限制

- 当前工作区缺少本地 `antigravity_core` 二进制，无法在本次实现阶段完成本机实测探针。
- 因此探测逻辑已落地到运行时，待有真实 core 可执行文件后即可得到准确能力报告。
